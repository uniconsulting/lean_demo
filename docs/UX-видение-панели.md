Родительские документы: `ТЗ-1-система-ЛЁН-MVP.md`, `ТЗ-2-прототип-стенд-и-стойка-автономии.md`, `домен-модель-и-контракты.md`, `план-этапов-проекта.md`, `docs/справочник/`. Конвенции: `AGENTS.md`.

Визуальная и компонентная спецификация вынесена в `UI-дизайн-система-панели.md`. Текущий этап: новый React golden screen «Сводка смены» в expressive industrial enterprise-направлении; остальные экраны переносятся только после его ручного утверждения.

## 1. UX-рамка

### 1.1 Главная идея интерфейса

ЛЁН не должен ощущаться как "панель мониторинга принтеров". Мониторинг является только слоем наблюдения. Главная ценность продукта для оператора и клиента:

```
изделие -> слайсинг -> серия -> очередь -> печать -> контроль -> снятие -> сортировка -> журнал -> отчет
```

Панель должна помогать ферме производить серии деталей с минимальным участием человека, а не просто показывать красивые карточки устройств. Поэтому главный язык интерфейса:
- производственный поток;
- готовность к ночи;
- отклонения от нормы;
- задачи оператора;
- трассируемость каждой детали.

### 1.2 Принципы Nielsen в применении к ЛЁН

| Эвристика | Как проявляется в прототипе |
|---|---|
| Видимость статуса системы | На любом экране видно: что печатает ферма, что заблокировано, что требует человека, готова ли ночь. |
| Соответствие реальному миру | Термины экрана соответствуют производству: изделие, серия, катушка, тара, смена, годен/брак, стойка, принтер. |
| Контроль пользователя | Оператор может подтвердить/опровергнуть инцидент, изменить очередь, отложить задачу, открыть первоисточник события. |
| Предотвращение ошибок | Деструктивные действия требуют подтверждения; ночной запуск не проходит при блокерах; ABS/ASA не получает авто-паузу. |
| Консистентность | Одни и те же статусы, блокеры, badges и action-панели повторяются на всех экранах. |
| Узнавание вместо запоминания | Правила ночи, причины блокеров, требуемые действия и последствия отображаются рядом с действием. |
| Гибкость | Desktop/tablet покрывает основную работу; PWA/Telegram дают быстрые подтверждения и сводки. |
| Минимализм | На экране сначала видны статус, риск, следующее действие; детали раскрываются по необходимости. |
| Восстановление после ошибок | Любая ошибка показывает: что случилось, какой объект затронут, что можно сделать дальше, кто уже подтверждал. |
| Помощь и документация | В прототипе помощь не как отдельная "справка", а как короткие подсказки прямо в блокерах, задачах и onboarding. |

### 1.3 Lean/Kaizen/Imai в применении к ЛЁН

| Lean-паттерн | UX-решение |
|---|---|
| Visual management | Карта парка, очередь, ночь и задачи показывают норму/отклонение без чтения длинных логов. |
| Standard work | День оператора разложен на утро, день, вечер; повторяемые процедуры оформлены как чек-листы. |
| Andon | Любое отклонение превращается в видимый сигнал: инцидент, задача, блокер ночи, fault-состояние принтера. |
| Jidoka | Система останавливает или эскалирует проблему до продолжения дефектного потока; человек подтверждает опасные решения. |
| Flow | Основная навигация ведет от изделий и очереди к выпуску деталей, а не к ручному управлению устройствами. |
| Muda reduction | Оператору показываются только действия, которые реально разблокируют поток: катушка, тара, поверхность, инцидент, preflight. |
| Gemba | Экран всегда связан с физическим объектом: принтер, стойка, тара, катушка, деталь, кадр камеры. |
| Continuous improvement | Подтверждения/опровержения инцидентов и результаты чекпоинтов становятся датасетом для улучшения CV/VLM. |

## 2. Персоны и день оператора

### 2.1 Оператор фермы

Главная персона MVP. Работает с панелью как с операционной системой смены.

Цели:
- быстро понять, что произошло ночью;
- не искать проблемы вручную по камерам и логам;
- ставить серии из библиотеки;
- выполнять только нужные задачи;
- к вечеру получить понятный ответ: можно ли оставить стойку на 12 часов.

Ограничения:
- оператор не обязан понимать внутренности CV/VLM, edge/center и MQTT;
- оператор не должен помнить правила материалов и ночных блокеров;
- интерфейс должен объяснять, почему действие разрешено или запрещено.

### 2.2 Администратор фермы

В MVP вторичная персона. Работает с отчетами, номенклатурой, ролями и результатами смен.

Цели:
- видеть выпуск, брак, загрузку и причины простоев;
- управлять допусками изделий к серии и ночной смене;
- контролировать пользователей и права;
- выгружать сменные отчеты.

### 2.3 Инженер поддержки ЛЁН

В MVP вторичная персона. Подключается через WireGuard и смотрит диагностику.

Цели:
- видеть состояние edge/center, синхронизацию, деградацию связи;
- смотреть историю fault-состояний, инцидентов и команд;
- помогать оператору без прямого вмешательства в физический цикл, если это не разрешено ролью.

### 2.4 Мобильный пользователь PWA/Telegram

Не полноценная замена desktop-панели. Мобильный сценарий MVP:
- утренняя сводка;
- срочный алерт;
- подтверждение или отказ деструктивного действия;
- быстрый просмотр задачи.

На мобильном в MVP не делаем:
- сложную расстановку очереди drag&drop;
- onboarding изделия;
- настройку профилей;
- анализ длинных отчетов;
- полноценное управление картой парка.

## 3. Операционный цикл

### 3.1 Утро: разобрать ночь

Сценарий:
1. Оператор открывает панель и видит сводку ночной смены.
2. Система показывает выпуск, брак, инциденты, вмешательства, простои, состояние тары и расходников.
3. Оператор последовательно закрывает задачи: забрать тару, заменить катушку, помыть/заменить поверхность, подтвердить инцидент, долить клей, если ветка адгезии по Э1 требует клей-модуль 🔬.
4. По закрытым задачам журнал смены становится готов к отчету.

UX-цель: оператор не начинает день с хаотичного просмотра камер. Он начинает с "что требует моего действия".

### 3.2 День: управлять сериями

Сценарий:
1. Оператор выбирает изделие из библиотеки или заводит новое.
2. Для нового изделия проходит flow: модель -> профиль -> слайсинг -> калибровочные печати -> эталоны первого слоя -> допуск к серии/ночи.
3. Для серийного изделия оператор ставит задание в очередь.
4. Очередь предсказуемо распределяется диспетчером по свободным совместимым принтерам.
5. Инциденты и задачи приходят в течение дня как andon-сигналы.

UX-цель: оператор управляет производством через изделия и серии, а не через ручной запуск отдельных принтеров.

### 3.3 Вечер: зарядить ночь

Сценарий:
1. Оператор открывает экран "Зарядить ночь".
2. Система считает готовность на 12 часов: очередь, материалы, расходники адгезии, тара, исправность принтеров, preflight, пожарная цепь, связь/edge.
3. Если все готово, оператор видит состояние "ночь готова" и подтверждает запуск ночного режима.
4. Если есть блокеры, экран показывает конкретный объект, причину и действие: заменить катушку #7, очистить тару брака, исключить принтер K1C-04, добавить задания на 3 часа.
5. Ночью сомнительные preflight-ситуации не будят человека: принтер пропускается, если политика ТЗ это требует.

UX-цель: "ночь" становится управляемым производственным протоколом, а не надеждой, что все само доедет.

## 4. Информационная архитектура

### 4.1 Основная навигация MVP

1. **Сводка**: текущий статус фермы, ночь, задачи, инциденты, выпуск.
2. **Парк**: стойки, принтеры, камеры, состояния FSM.
3. **Изделия**: библиотека, версии, профили, допуски, эталоны.
4. **Очередь**: серийные задания, порядок, совместимость, блокеры.
5. **Инциденты**: кадры, VLM-вердикты, действия человека, датасет.
6. **Задачи**: работа оператора, регламенты, обслуживание, расходники.
7. **Журнал**: детали, трассируемость, исходы годен/брак.
8. **Отчеты**: смена, CSV, KPI, причины простоев.
9. **Настройки**: роли, пользователи, уведомления, принтеры, профили.

### 4.2 Глобальная раскладка low-fi

Wireframe-стиль: нейтральная монохромная основа без финальной графики. После выбора референсного макета 11 июля 2026 закреплена более плотная операторская компоновка ✅:
- узкая левая навигация;
- верхняя KPI-полоса с состоянием фермы и инфраструктуры;
- центральная карта/рабочая область;
- постоянная правая колонка следующего действия, инцидента и готовности ночи;
- нижний журнал/техническая строка там, где они поддерживают сценарий.

Базовая структура desktop/tablet:
- левая вертикальная навигация;
- верхняя строка статуса: площадка, связь edge/center, текущая смена, ночной режим, критичные алерты;
- центральная рабочая область;
- правая контекстная панель: задачи, выбранный объект, блокеры, история действий;
- нижняя строка/панель состояния на экранах с потоками: очередь, прогресс, last sync, pending actions.

Обязательные повторяемые элементы:
- статус-бейдж объекта;
- "следующее действие";
- причина блокера;
- источник решения: правило / CV / VLM / оператор / hardware;
- отметка синхронизации edge -> center;
- ссылка на журнал/аудит события.

### 4.3 Геометрия прототипов

Прототипы строятся не через обводки, а через различие поверхностей. Border для фреймов и блоков по умолчанию не используется; допустимы только технические разделители внутри таблиц/логов, если без них теряется читаемость.

Палитры темы ✅:
- ночная: нейтральный графит и темно-серые поверхности без синего подтона; бело-серый текст;
- светлая: белые и холодные светло-серые поверхности без кремового подтона;
- зеленый, янтарный, красный и информационный голубой используются только семантически — состояние, ожидание, ошибка, переход — и не окрашивают базовые фреймы.

Иконография и переходы ✅:
- единая библиотека Lucide; базовый размер навигации `16 px`, stroke около `1.7 px`;
- номера разделов в навигации не используются — каждый раздел имеет устойчивую предметную метафору;
- тема применяется до первого кадра страницы, выбор сохраняется между разделами;
- оболочка панели не пересоздается при смене разделов: новый маршрут заменяет только рабочую область, сохраняя URL и browser history;
- переходы короткие и ненавязчивые, анимация ограничена изменяемым контентом и обязательно учитывает `prefers-reduced-motion`.

Базовый ритм:
- primitive: 4 px;
- major scale: 8 px;
- radii, spacing и shell-размеры задаются через adaptive `clamp()`.

Правило вложенных радиусов:

```text
R2 = R1 - I
```

Где:
- `R1` — radius ближайшего смыслового внешнего фрейма;
- `I` — inset / внутренний отступ от внешнего фрейма до вложенного;
- `R2` — radius вложенного фрейма.

После выбора более плотного референса внешний radius задается диапазоном `18–24 px` через `clamp()` ✅. Пример: внешний модуль `24 px`, inset `12 px`, вложенный radius `12 px`. Визуальная цель: углы вложенного блока продолжают ту же оптическую дугу, что и внешний контейнер.

Ключевое правило: radius считается от ближайшего смыслового внешнего фрейма, а не от всей страницы. Например, поле поиска внутри модуля "Чаты" наследуется от фрейма модуля "Чаты", а не от общего фрейма раздела.

Auth-исключения, если появится auth-flow:
- auth image frame: около `74 px` на ширине `1440`;
- auth card: `36 px`;
- input radius: `18 px`;
- input interval: `18 px`.

## 5. Карта MVP-экранов

### 5.1 Сводка смены

Цель: дать оператору один стартовый экран вместо ручной проверки всех разделов.

Пользовательский сценарий:
- утром оператор видит итоги ночи;
- днем видит текущий поток;
- вечером видит, насколько ферма близка к готовности ночи.

Wireframe-блоки:
- верхняя строка: текущая смена, режим, связь, критичный статус;
- крупный блок "сейчас": печатают / ждут / fault / контроль;
- блок "выпуск": деталей годен, брак, кг/шт, выполнение очереди;
- блок "инциденты": новые, ожидают подтверждения, критичные;
- блок "задачи оператора": что сделать первым;
- блок "ночь": готовность, часы очереди, число блокеров;
- блок "последние события": отмены, паузы, сбросы, синхронизация.

Основные состояния:
- нормальный день;
- утро после успешной ночи;
- утро с инцидентами;
- центр недоступен, edge продолжает;
- пожарный/критичный алерт.

Действия:
- перейти к задачам;
- открыть инцидент;
- открыть "Зарядить ночь";
- открыть отчет смены;
- подтвердить прочтение критичной сводки.

Блокеры и ошибки:
- нет данных от edge;
- есть несинхронизированный журнал;
- есть инциденты без решения человека;
- есть задачи, блокирующие ночной запуск.

Backend-сущности и события:
- сущности: ShiftReport, Printer, Job, Incident, OperatorTask, PartInstance;
- команды/события: acknowledge incident, mark task done, export report, sync backlog status.

### 5.2 Карта парка

Цель: показать физическую картину фермы и отклонения от нормы.

Пользовательский сценарий:
- оператор сканирует стойки и принтеры;
- видит, какие устройства автономные, какие только "Контроль";
- быстро переходит в карточку проблемного принтера.

Wireframe-блоки:
- фильтры: стойка, режим, материал, статус, только блокеры;
- сетка стоек;
- карточка принтера: миниатюра камеры, FSM-статус, материал, изделие, ETA, риск/блокер;
- отдельный блок Bambu/Control-only: камера + реле + учет, без команд прошивки;
- правая панель выбранного принтера.

Основные состояния:
- все в норме;
- принтер печатает;
- preflight не уверен, печать не стартует;
- принтер в fault;
- control-only без телеметрии;
- offline/нет камеры.

Действия:
- открыть карточку принтера;
- отфильтровать блокеры;
- вручную отметить занятость control-only принтера;
- открыть live-кадр;
- отправить безопасную команду, если роль разрешает.

Блокеры и ошибки:
- камера недоступна;
- Moonraker недоступен;
- принтер не прошел preflight;
- mismatch: материал задания не совпадает с катушкой;
- поверхность/клей превысили лимит 🔬;
- пожарная цепь не прошла heartbeat.

Backend-сущности и события:
- сущности: Printer, Rack, Site, Spool, Job, Incident, OperatorTask;
- команды/события: update printer state, relay power request, manual busy mark, open printer details, create operator task.

### 5.3 Карточка принтера

Цель: дать полный контекст одного принтера без перегруза карты парка.

Пользовательский сценарий:
- оператор открывает принтер из карты или инцидента;
- видит live-камеру, телеметрию, текущее задание, историю;
- понимает, какое действие безопасно и что заблокировано.

Wireframe-блоки:
- live-камера/последний кадр;
- FSM-лента: idle -> preflight -> heating -> printing -> cooling -> ejecting -> sorting -> confirm_drop;
- телеметрия: температуры, прогресс, ETA, связь;
- текущее задание и изделие;
- катушка и остаток;
- поверхность/клей и счетчики 🔬;
- последние события и инциденты;
- action-панель с разрешенными командами.

Основные состояния:
- idle и готов к заданию;
- printing с чекпоинтом первого слоя;
- cooling/ejecting/sorting;
- waiting_human;
- fault механики;
- center unavailable, локальный режим edge.

Действия:
- открыть задание;
- открыть изделие;
- подтвердить/отклонить cancel;
- создать задачу обслуживания;
- заменить катушку;
- отправить pause/stop только с подтверждением и правами.

Блокеры и ошибки:
- деструктивное действие требует подтверждения;
- ABS/ASA: авто-пауза запрещена, только алерт;
- рассогласование ИК/весов после сброса;
- перегрев/дым переводит в локальный safety state;
- нет данных центра, но edge продолжает.

Backend-сущности и события:

### 5.5 Onboarding изделия

Цель: превратить новый STL/3MF в управляемое производственное изделие.

Пользовательский сценарий:
1. Оператор загружает модель.
2. Выбирает связку принтер/материал/поверхность или профиль.
3. Запускает серверный слайсинг.
4. Система создает калибровочные печати.
5. Первые слои становятся эталонами.
6. Изделие получает допуски к серии и затем к ночи.

Wireframe-блоки:
- stepper: модель -> профиль -> слайсинг -> калибровка -> эталоны -> допуск;
- зона файла и метаданных;
- выбор профиля;
- результаты слайсинга: время, вес, материал, макросы post-processing;
- список калибровочных печатей;
- эталоны первого слоя;
- финальное решение допуска.

Основные состояния:
- файл загружен;
- slicing queued/running/failed;
- calibration prints running;
- awaiting human review;
- approved/blocked.

Действия:
- загрузить STL/3MF;
- выбрать профиль;
- запустить слайсинг;
- повторить слайсинг;
- назначить калибровочную печать;
- принять/отклонить эталон;
- выдать допуск.

Блокеры и ошибки:
- неподдерживаемый файл;
- профиль не найден;
- Orca CLI вернул ошибку;
- post-processing не вставил обязательные макросы;
- нет эталонов первого слоя;
- калибровочные печати дали нестабильный результат.

Backend-сущности и события:
- сущности: Product, SliceArtifact, Printer, Spool, Job, Incident;
- команды/события: upload model, slice request, slice completed, slice failed, calibration job created, first layer reference saved, approve product.

### 5.7 Очередь серий

Wireframe-блоки:

Backend-сущности и события:

### 5.8 Инциденты

### 5.10 Зарядить ночь

Цель: главный MVP-сценарий панели. Дать честный ответ: можно ли оставить стойку на 12 часов без человека.

Пользовательский сценарий:
- оператор открывает экран вечером;
- видит агрегированный чек-лист готовности;
- закрывает блокеры;
- подтверждает ночной режим или осознанно исключает проблемные принтеры.

Wireframe-блоки:
- крупный статус: ночь не готова / готова с предупреждениями / готова / активна;
- прогноз очереди: часов работы, заданий, материалов;
- чек-лист:
  - очередь >= 12 ч;
  - филамента хватает;
  - тары пусты;
  - расходники адгезии в норме 🔬;
  - автономные принтеры прошли preflight;
  - пожарная цепь живая;
  - edge готов, зеркало очереди и G-code есть;
  - центр доступен или деградация допустима;
- список блокеров с объектами и действиями;
- кнопка approve night;
- ночная политика: что будит, что не будит, что пропускается.

Основные состояния:
- not_ready;
- ready_with_warnings;
- ready;
- active;
- degraded;
- finished.

Действия:
- approve night;
- exclude printer from night;
- add jobs;
- replace spool;
- mark bin emptied;
- run preflight;
- open blocker;
- cancel night mode.

Блокеры и ошибки:
- очереди меньше 12 ч;
- задание требует больше филамента, чем остаток катушки;
- тара годен/брак не пуста или весы не подтверждают;
- принтер fault/offline;
- поверхность/клей превысили лимит 🔬;
- пожарная цепь не подтверждена;
- центр недоступен >= 24 ч: новые задания не стартовать без локального подтверждения.

Backend-сущности и события:
- сущности: Job, Printer, Spool, BuildSurface/GlueUnit, OperatorTask, ShiftReport, Incident;
- команды/события: approve night, exclude printer, run preflight, mark bin emptied, replace spool, queue coverage calculated, degradation policy applied.

## 6. Уведомления и подтверждения

### 6.1 Каналы MVP

| Канал | Назначение |
|---|---|
| Desktop/tablet панель | Основная работа: очередь, изделия, карта, задачи, отчеты, инциденты. |
| PWA | Быстро посмотреть статус, задачу, инцидент, подтвердить действие. |
| Telegram | Временный мобильный канал MVP: алерты, утренняя сводка, подтверждение деструктива. |

### 6.2 Типы уведомлений

| Тип | Пример | Канал | UX-правило |
|---|---|---|---|
| Информационное | смена завершена, отчет готов | панель, PWA | Не прерывает работу. |
| Задача | заменить катушку, очистить тару | панель, PWA | Показывает объект и действие. |
| Блокер ночи | не хватает очереди, тара не пуста | панель | Ведет прямо в экран "Зарядить ночь". |
| Инцидент | VLM просит подтвердить cancel | панель, Telegram/PWA | Требует явного решения человека. |
| Safety | дым, питание, перегрев | все доступные | Показывает локальное действие системы и что уже обесточено. |
| Деградация | центр недоступен, edge продолжает | панель | Объясняет, что продолжает работать и что ограничено. |

### 6.3 Подтверждение деструктивных действий

Деструктивные действия: cancel, stop, power off, исключение принтера из ночи, корректировка исхода детали, закрытие отчета при незакрытых инцидентах.

Паттерн подтверждения:
- действие формулируется как итог: "Отменить печать задания #123 и поставить перепечатку";
- показывается объект, причина и источник рекомендации;
- показываются последствия: деталь уйдет в брак, задание вернется в очередь, событие попадет в аудит;
- кнопки: подтвердить / отклонить / открыть детали;
- после действия показывается результат, а не просто исчезновение modal.

| mark bin emptied | Задачи, ночь |
| retry print | Инцидент, очередь, журнал |
| export report | Отчет смены, журнал |
| run preflight | Ночь, карточка принтера |
| exclude printer from night | Ночь |
| create maintenance task | Карточка принтера, инцидент |
| sync backlog status | Сводка, журнал, деградация |

### 7.3 Поля, которые UX ожидает от backend без фиксации DTO

Для каждого экранного объекта нужны:
- `id` и человекочитаемое имя;
- статус и причина статуса;
- источник статуса: rule, operator, CV, VLM, hardware, edge, center;
- timestamp последнего события;
- связь с физическим объектом: site/rack/printer/bin/spool;
- next_action, если объект требует человека;
- audit trail для деструктивных действий;
- sync_state, если запись родилась на edge и еще не ушла в центр.

## 8. Открытые решения и границы MVP

Не выдумывать в UX решения, которые в ТЗ помечены открытыми или экспериментальными.

| Вопрос | Как показывать в прототипе |
|---|---|
| Состав парка ФТ ❓ | В wireframe использовать абстрактные стойки и K1-принтеры; точные модели подставляются после аудита. |
| Модель стойки автономии ❓ | Показывать 4-8 принтеров как диапазон; не фиксировать конкретную раскладку. |
| CF-материалы ❓ | Не делать отдельный UX-поток; оставить как материал, который может появиться после аудита. |
| Камеры фермы ❓ | UX требует live/last frame и статус камеры; тип USB/IP/PoE не фиксировать. |
| Размещение центра ❓ | Показывать состояние center/edge и правила деградации; физическое размещение не фиксировать. |
| Адгезия клей vs PEI 🔬 | В UX заложить абстрактный блок "поверхность/клей"; финальные поля зависят от Э1. |
| Web-preview слоев | Этап 2; в MVP показывать только метаданные slice artifact и предупреждения. |
| LLM-рекомендации параметров | Этап 2; в MVP VLM только разбирает инциденты. |
| Нативное мобильное приложение | Вне MVP; PWA/Telegram достаточно. |

## 9. Прототипные маршруты для Canva

Эти маршруты нужны как кликабельная логика low-fi прототипа.

1. **Утро после ночи**: Сводка -> Инциденты -> Задачи -> Журнал -> Отчет смены.
2. **Поставить серию**: Библиотека -> Изделие -> Создать задание -> Очередь -> Карта парка.
3. **Завести изделие**: Библиотека -> Onboarding -> Слайсер-поток -> Калибровка -> Допуск.
4. **Разобрать инцидент**: Сводка/Telegram -> Инцидент -> Карточка принтера -> Подтверждение -> Журнал.
5. **Сменить катушку**: Задача -> Карточка принтера -> Replace spool -> Весы/остаток -> Очередь разблокирована.
6. **Зарядить ночь без блокеров**: Сводка -> Зарядить ночь -> Все checks ok -> Approve night -> Ночной режим active.
7. **Зарядить ночь с блокерами**: Зарядить ночь -> Блокер катушки/тары/очереди -> Задача -> Повторная проверка -> Approve night.
8. **Обрыв связи центра**: Сводка -> Degraded banner -> Очередь edge mirror -> Журнал sync backlog -> Восстановление.
9. **ABS/ASA-инцидент**: Инцидент -> Политика "без авто-паузы" -> Алерт человеку -> Решение в журнал.

## 10. Acceptance checklist для UX-документа

### 10.1 Nielsen-check

- На каждом экране видно текущее состояние системы.
- Главное действие экрана не нужно искать.
- Опасное действие невозможно выполнить без подтверждения.
- Ошибка объясняет причину и следующий шаг.
- Оператору не нужно помнить правила материалов, ночи и деградации.
- Один и тот же статус называется одинаково во всех разделах.

### 10.2 Lean-check

- Экран помогает производственному потоку, а не просто показывает данные.
- Отклонение от нормы видно как andon-сигнал.
- Каждая задача оператора имеет физический объект и понятный результат.
- Ночной чек-лист работает как standard work.
- Подтверждения/опровержения инцидентов улучшают датасет.
- Нет ручных действий, которые система уже может вывести из данных.

### 10.3 Сценарный тест

Документ считается готовым к следующей итерации прототипа после ручного прохода по сценариям:
- оператор утром разбирает ночную смену;
- оператор ставит серию из библиотеки;
- оператор заводит новое изделие;
- система показывает инцидент с VLM-вердиктом;
- оператор подтверждает или опровергает решение;
- оператор меняет катушку;
- оператор готовит ночную смену;
- система показывает блокеры ночи;
- система работает при обрыве связи с центром.

Статус этого документа остается **нужен тест** до ревью Д + Ф по экранным потребностям и до сверки с `домен-модель-и-контракты.md`.
