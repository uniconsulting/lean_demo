## 2.1 Прошивки: Marlin vs Klipper

**Прошивка** — программа, исполняющая G-code и управляющая железом принтера (моторы, нагрев, датчики, защиты).

**Marlin** — классика: вся логика в одном микроконтроллере принтера. Настройки зашиты при компиляции, снаружи доступен только USB-serial (одна команда за раз). Для сетевого управления нужен посредник (OctoPrint на Raspberry Pi). В нашем MVP Marlin-машины не участвуют.

**Klipper** — двухчастная архитектура, и это его суперсила:
- **klippy** — «мозг» на Linux-компьютере (в K1-семействе он встроен прямо в принтер) — считает всю математику движения;
- **MCU-прошивка** — «спинной мозг» в микроконтроллере — тупо исполняет готовые команды с точными таймингами.

Что это даёт:
- **Конфиг вместо перекомпиляции**: весь принтер описан текстовым `printer.cfg` (пины, геометрия, лимиты, датчики). Правка = редактирование файла + перезапуск.
- **Макросы** — именованные программируемые последовательности G-code с логикой (шаблоны Jinja2: переменные, условия). Наши `COOLDOWN_EJECT`, `FIRST_LAYER_CHECKPOINT`, `WIPE`, `PARK` — это макросы в конфиге, которые система вызывает по имени. Есть `delayed_gcode` (отложенный запуск) и `save_variables` (переменные, переживающие перезагрузку).
- **Штатные фичи качества**: pressure advance (компенсация инерции экструзии), input shaping (гашение резонансов рамы — нужен акселерометр), thermal runaway protection (см. блок 6), bed mesh.
- **Датчики как события**: filament runout/motion sensor вызывают G-code при срабатывании.

**Root на K1-семействе**: с завода Creality ограничивает доступ; официальная процедура root открывает SSH и полный Moonraker. После root это обычный Klipper-принтер под нашим контролем. Риск: обновление заводской прошивки может перезаписать доступ и конфиги — поэтому на пилоте версии прошивок парка фиксируются (ТЗ-1, риски).

## 2.2 Moonraker — API-дверь в Klipper

**Moonraker** — сервер рядом с klippy, дающий HTTP + WebSocket API. Всё стороннее управление Klipper-принтером идёт через него:
- **запросы**: статус, температуры, прогресс печати, список файлов;
- **подписки** (WebSocket): принтер сам шлёт изменения объектов (`print_stats`, `toolhead`, `extruder`, `heater_bed`…) — не надо опрашивать в цикле;
- **команды**: загрузить G-code файл, старт/пауза/отмена, исполнить любой G-code или макрос (`COOLDOWN_EJECT`), управление питанием;
- **события**: filament runout, ошибки, завершение печати.

Веб-морды **Fluidd** и **Mainsail**, которые видно в браузере у K1C — просто клиенты Moonraker. **Наша система — ещё один клиент Moonraker, только один на весь парк**: edge-узел держит WebSocket-соединение с каждым принтером.

## 2.3 Слайсеры и профили

Родословная важна для понимания рынка: почти все современные слайсеры — форки одного дерева (Slic3r → **PrusaSlicer** → **Bambu Studio** → **OrcaSlicer**). Orca — открытый (AGPLv3), самый фермо-дружелюбный, есть **CLI** (запуск без GUI) — поэтому он крутится у нас на сервере как воркер: «вход: модель + профиль → выход: G-code».

**Профиль** — это три слоя настроек:
- **принтера** (геометрия, лимиты, стартовый/финишный G-code);
- **филамента** (температуры, обдув, ретракты, pressure advance — под конкретный материал и даже бренд);
- **процесса** (высота слоя, стенки, заполнение, скорости, суппорты).

Профили — половина качества печати и **наша накапливаемая ценность**: выверенная связка «принтер × материал × поверхность» = повторяемость серии. Библиотека профилей версионируется (изменил профиль — новая версия, старые детали печатались старой: трассируемость).

**Пост-процессинг G-code** — обработка файла после слайсера: наша система вставляет туда чекпоинт первого слоя и макросы автоснятия. Это дешёвый и мощный приём: мы «дописываем» поведение в любую печать, не трогая слайсер.

## 2.4 G-code: минимум для чтения

Текст, по команде на строку. Ключевые:

| Команда | Смысл |
|---|---|
| `G28` | Хоуминг — поиск нулей осей концевиками |
| `G0/G1 X.. Y.. Z.. E.. F..` | Движение (E — сколько выдавить, F — скорость) |
| `M104 S220` / `M109 S220` | Нагреть сопло / нагреть и ЖДАТЬ |
| `M140 S60` / `M190 S60` | То же для стола |
| `M106 S255` / `M107` | Обдув вкл (0–255) / выкл |
| `G29` / `BED_MESH_CALIBRATE` | Промер стола |
| `M600` | Смена филамента (пауза с выгрузкой) |
| `PAUSE` / `RESUME` / `CANCEL_PRINT` | Управление печатью (Klipper-макросы) |
| `SET_PRESSURE_ADVANCE`, `SET_VELOCITY_LIMIT` | Тюнинг на лету |
| `ИМЯ_МАКРОСА PARAM=X` | Вызов нашего макроса с параметрами |

Стартовый G-code (в профиле принтера) греет, хоумит, промеряет стол, рисует продувочную линию; финишный — гасит нагрев, паркует голову. Наши вставки живут точно там же.

## 2.5 Архитектура системы ЛЁН (карта, детали в ТЗ-1 разд. 4)

```
ЦЕНТР (наш сервер, GPU): VLM-вердикты · тяжёлая CV · слайсер-воркеры (Orca CLI)
                          PostgreSQL · веб-панель (React SPA/PWA) · API
        ↕ WireGuard (площадка соединяется исходящим туннелем, портов наружу нет)
EDGE (миникомп на площадке): КОНЕЧНЫЙ АВТОМАТ + ДИСПЕТЧЕР · MQTT-брокер
                          зеркало очереди + кэш G-code · локальная CV · видеобуфер · реле
        ↕ LAN
ПРИНТЕРЫ: Klipper/Moonraker (WebSocket) · камеры · ESP32-механика (MQTT) · датчики дыма/реле
```

Принципы, которые надо держать в голове при любой разработке:
1. **Детерминированное ядро без LLM**: конечный автомат принтера, диспетчер очереди, вся безопасность — обычный код. LLM (VLM-вердикт) — единственная ИИ-точка, асинхронная, с гард-рейлами и аудитом.
2. **Edge автономен**: обрыв интернета не останавливает производство (правила деградации в ТЗ-1). Поэтому автомат и диспетчер живут на площадке, а центр — надстройка.
3. **Интерфейсы вместо технологий**: `EventBus`, `Notifier`, `SliceService` — ядро не знает, что под ними (сегодня MQTT+Postgres и Telegram, завтра NATS и мобильная прила).
4. **Каскад CV**: дешёвая модель смотрит всегда → тяжёлая проверка по подозрению → VLM-вердикт → действие. Исключение — чекпоинт первого слоя: там тяжёлая проверка КАЖДОЙ печати (событие редкое, можно позволить).

## 2.6 Конечный автомат печати (упрощённо)

```
idle → preflight (стол пуст? материал? лимиты?) → heating → printing
     → cooling (обдув до T отлипа) → ejecting (дверь, сниматель)
     → sorting (заслонка по вердикту) → confirm_drop (ИК + весы) → idle
```
Каждый переход — событие в журнале. Отклонение (деталь не снялась, вес не сошёлся) — ошибочная ветка: стоп + задача человеку. Полная схема — `docs/домен-модель-и-контракты.md`.

## 2.7 CV и ML в проекте — словарь

- **YOLO** — семейство быстрых детекторов: кадр → рамки «спагетти 0.72», «стол не пуст 0.95». Лёгкие версии (n/s) крутятся на edge, крупные — на центре.
- **VLM** (vision-language model, у нас Qwen2.5-VL) — мультимодальная LLM: понимает кадр в контексте («это суппорты, а не спагетти»), отвечает структурированным вердиктом. Медленная (секунды) — только эскалации и чекпоинты.
- **PatchCore/PaDiM** — детекция аномалий: учится на фото «нормы» (эталонные первые слои изделия) и флажит любое отклонение. Разметка брака не нужна — идеально для серийки.
- **ONNX** — переносимый формат моделей: один файл гоняется и на GPU центра (TensorRT), и на CPU edge (OpenVINO). Наше правило: один артефакт, автотест паритета предсказаний.
- **Датасет** — наша копилка: каждый инцидент + вердикт + «человек подтвердил/опроверг» = обучающий пример. Катастрофы печатаются намеренно на стенде (Э6) — дёшево и с чистой разметкой.
