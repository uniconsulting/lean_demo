# UI-дизайн-система панели ЛЁН

Статус: **нужен тест**. Golden screen «Сводка смены» реализован; перенос остальных экранов начнётся после ручного визуального и сценарного ревью.

Родительские документы: `ТЗ-1-система-ЛЁН-MVP.md`, `UX-видение-панели.md`, `домен-модель-и-контракты.md`. Реализация токенов: `src/styles/tokens.css`.

## 1. Направление

ЛЁН использует выразительный industrial enterprise-язык: рабочий экран строится вокруг физического объекта фермы и следующего действия оператора, а не вокруг сетки равнозначных KPI-карточек.

Визуальный ориентир — публичные работы [RON Design Lab](https://www.instagram.com/rondesignlab/). В проект переносится не конкретный интерфейс и не сторонние изображения, а общие паттерны: объектный фокус, асимметрия, крупная типографика, display-метрики, микро-визуализации и ограниченное число цветных поверхностей.

- 70–80% площади занимают нейтральные белые/холодно-серые или графитовые поверхности.
- 20–30% занимают семантические signal-модули.
- Крупный объект показывает физический контекст: стойку, принтер или изделие.
- Цвет всегда дублируется текстовым статусом, иконкой или формой маркера.
- Blur допустим только как фон крупной акцентной поверхности; рабочий текст остаётся контрастным.
- Signal-поверхности используют широкие многотональные диагональные поля без локального кругового блика: переход должен ощущаться как рассеянный свет, а не как отдельная светящаяся точка.
- Основные фреймы разделяются поверхностями, тенью и отступом, без CSS `border`.

## 2. Токены

### 2.1 Цвет и темы

Светлая тема основана на `#eef0f1` для canvas, белых основных поверхностях и холодных серых вложенных блоках. Ночная тема использует нейтральный graphite `#101111`, без тёмно-синего подтона.

| Роль | Светлая тема | Ночная тема |
|---|---|---|
| Canvas | холодный светло-серый | нейтральный почти чёрный |
| Surface | белый | graphite 1 |
| Nested surface | холодный серый | graphite 2 |
| Primary text | почти чёрный | почти белый |
| Secondary text | средний серый | светлый серый |
| Stable | зелёно-лаймовый | зелёный полупрозрачный градиент |
| Action | янтарно-оранжевый | янтарный полупрозрачный градиент |
| Incident | красно-коралловый | красный полупрозрачный градиент |
| Information | холодный голубой | холодный голубой полупрозрачный градиент |

Тень основных фреймов в светлой теме намеренно слабее. Тонкие технические линии формируются внутренним `box-shadow`, а не обводкой; в светлой теме их интенсивность ниже, чем в ночной.

### 2.2 Типографика

- Интерфейс: variable grotesk **Manrope**, локально включённый через `@fontsource`.
- Ключевые числа: dotted-display **Doto**. Он используется только для производственных метрик и не заменяет основной шрифт.
- Заголовки строят иерархию размером и воздухом, а не декоративными разделителями.
- Табличные числа используют `font-variant-numeric: tabular-nums`.

### 2.3 Геометрия и ритм

- Primitive rhythm: `4px`; основной шаг: `8px`.
- Основные радиусы: `18–24px`, адаптивные через `clamp()`.
- Вложенный радиус: `R2 = R1 - I`, где `I` — inset от ближайшего смыслового внешнего фрейма.
- Размеры sidebar, context rail, spacing и shell рассчитываются через адаптивные токены.
- Плотность данных меняется брейкпоинтами, но смысловой порядок блоков не меняется.

## 3. Компонентная модель

### Primitives — `src/components/ui/`

- `Surface` — нейтральная или signal-поверхность с единым radius/shadow contract.
- `DottedNumber` — display-метрика с доступным текстовым значением.
- `DottedTime` — время с отдельно центрированным разделителем, чтобы особенности display-шрифта не смещали двоеточие.
- `LiveDot` — единый recording-индикатор активного процесса: одна точка без внешнего halo, плавно меняющая opacity.
- `RouteSkeleton` — локальный fallback нового маршрута без snapshot старой страницы.

### Производственные компоненты — `src/components/domain/`

- `FarmObject` — объектная карта стойки с доступными статусами и выбором принтера.
- Следующими выделяются `PrinterObject`, `ProductObject`, `StatusConstellation`, `ActionSurface`, `IncidentSurface` и `TraceTimeline` после подтверждения golden screen.

### Shell — `src/app/`

`AppShell`, Sidebar, CommandBar, RouteStage, ContextRail и ThemeProvider не размонтируются при переходе. Маршрут заменяет только содержимое RouteStage. Навигация использует Lucide React; специальные схемы оборудования остаются отдельными assets без UI-текста.

Правая часть CommandBar строится справа налево: theme control → профиль того же размера → имя сотрудника → вертикальный разделитель → статус edge. Размер кнопок и inset сверху, снизу и справа одинаковы. На touch-компоновке вторичная информация скрывается, theme control остаётся.

Три производственные KPI-поверхности имеют одинаковую геометрию. Действие очереди представлено icon-only control в заголовке. Все три микро-графика используют один утверждённый rounded-bar язык, но получают разные последовательности высот, соответствующие разным показателям.

Базовые object-assets находятся в `public/assets/`: `farm-rack.svg`, `printer-k1.svg`, `selected-product.svg`. Это оригинальные нейтральные line-art схемы; они получают цвет из контекста и не содержат подписей или статусов.

## 4. Motion

- Появление только нового маршрута: `120ms`, opacity + небольшое смещение.
- Старый экран удаляется из DOM сразу при смене маршрута; snapshot старого экрана не создаётся.
- `startViewTransition` и `view-transition-old` для маршрутов запрещены.
- Смена темы и hover-состояния используют короткий UI-token без изменения геометрии.
- `prefers-reduced-motion: reduce` отключает необязательные перемещения и переходы.

## 5. Доступность

- Любой статус передаётся минимум двумя каналами: цвет + текст, цвет + иконка либо цвет + форма.
- Все интерактивные объекты доступны с клавиатуры и имеют видимый `:focus-visible`.
- У кнопок-иконок есть доступные имена; декоративная графика скрыта от accessibility tree.
- Основной текст не размещается поверх низкоконтрастного blur.
- Размер интерактивной цели не меньше 40px на desktop и 44px в touch-компоновке.

## 6. Сопоставление паттернов с экранами

| Экран | Главный объект | Выразительная поверхность | Главный паттерн |
|---|---|---|---|
| Сводка | стойка фермы | выпуск, действие, инцидент | состояние смены → следующий шаг |
| Ферма | физическая карта стоек | отклонение выбранной зоны | spatial overview |
| Принтер | K1 и текущая деталь | состояние/температура/fault | object detail |
| Очередь | поток серий | покрытие и конфликт | ranked flow |
| Ночь | стойка и preflight | готовность/блокер | safety checklist |
| Изделия | выбранная модель | допуск к серии/ночи | visual library |
| Onboarding | новая модель | текущий gate | guided sequence |
| Слайсер | изделие + build plate | validation result | processing flow |
| Инциденты | кадр/принтер | требование решения | evidence first |
| Задачи | физический объект задачи | приоритет/дедлайн | operator work queue |
| Журнал | выбранная деталь | trace status | immutable timeline |
| Отчёт | смена как интервал | выпуск/качество | narrative analytics |
| Роли | оператор | разрешение/обучение | capability matrix |

## 7. Gate переноса

До ручного подтверждения golden screen остальные маршруты являются архитектурным каркасом с переходами в legacy-прототип. Ревью должно подтвердить композицию, типографику, объектный фокус, цветовые акценты, геометрию, плотность, обе темы и понятность следующего действия. После этого компоненты переносятся пакетами, указанными в плане миграции.
