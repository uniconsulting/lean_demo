## 1. Назначение и рамка

Система ЛЁН — программно-аппаратный комплекс автоматизации ферм FDM-печати: от загрузки модели до готовой детали в таре, с контролем качества и минимальным участием персонала.

MVP несёт **два раздельных уровня ценности** ✅ — они разрабатываются параллельно и принимаются раздельно, провал или задержка одного не обнуляет второй:

1. **Софт-контур** (весь Klipper-парк площадки): библиотека изделий, серверный слайсинг, очередь серийных заданий, мониторинг, CV-детекция с чекпоинтом первого слоя, трассируемость, панель оператора. Самостоятельная ценность независимо от железа.
2. **Контур автономии** («стойка автономии», 4–8 принтеров): физический безлюдный цикл — ночная смена 12 часов без человека: печать → снятие → сортировка годен/брак → учёт → следующая печать.

Пилот — ферма ФлайТех (Москва). Продукт проектируется тиражируемым, но **все требования этого ТЗ сформулированы под пилот**; тиражные требования (коробочная установка, автопровижининг) — этап 2.

### Чем MVP НЕ является
- Не «умный мониторинг» — мониторинг и очередь Bambu раздаёт бесплатно (Farm Manager), Obico — open source. Ценность ЛЁН — уровнем выше: изделия→серия→безлюдный цикл→деталь в таре.
- Не мультиагентная LLM-платформа — в MVP ровно одна ИИ-точка принятия решений (VLM-разбор инцидентов, п. 5.4) поверх детерминированного ядра.
- Не собственный слайсер-движок — обвязка поверх открытого движка (п. 5.10).

## 2. KPI пилота (критерии приёмки MVP)

KPI **стенда** (MTBF снимателя, catch-rate на намеренном браке, время цикла) — в ТЗ-2; они закрываются ДО выезда на пилот и в эту таблицу не входят ✅.

| # | KPI | Целевое значение | Определения и измерение |
|---|---|---|---|
| 1 | Безлюдная ночная смена | 12 ч × 5 ночей подряд на стойке автономии | «Вмешательство» = любое физическое действие человека на стойке или команда, отменяющая решение системы (просмотр панели вмешательством не является). Все исправные принтеры стойки печатают; простой по внешней причине (кончился филамент по плану) — не срыв. Журнал + видеоархив |
| 2 | Ложные остановки | ≤1 на 50 печатей; на ABS/ASA авто-пауз ровно 0 | Прогон обязан содержать ≥100 печатей (обеспечивается серийной номенклатурой), иначе критерий не считается проверенным. Журнал пауз с причинами |
| 3 | Пропуск катастроф | 0 пропущенных инцидентов класса «спагетти / пожароопасность» за прогон | Сверка видеоархива с журналом инцидентов |
| 4 | Занятость оператора | 100% задач смены поступает из панели/бота; хронометраж занятости фиксируется | Формулировка через систему (лог задач + хронометраж), а не через штатное расписание клиента — кадровые решения ФТ вне нашего контроля |
| 5 | Трассируемость | 100% деталей в журнале: экземпляр ← задание ← изделие ← катушка ← принтер ← исход + фото | Выгрузка журнала vs физический пересчёт тары; при обрыве связи допускается отложенная синхронизация в центр (журнал первичен на edge, п. 5.5) |

Метрика контура первого слоя (п. 5.3.1) **измеряется с первого дня пилота, но в критерии приёмки MVP числом не входит** — числовой KPI назначается на этап 2 по фактическим данным ✅.

## 3. Объект автоматизации

- **Парк MVP: Creality K1-семейство** (K1C, K1/K2, K1 Max) — Klipper/Moonraker, root по официальной процедуре Creality, все модели закрытые ✅. Точный состав парка и выбор модели для стойки автономии (одна модель = меньше переменных) — аудит, веха 0. ❓
- **Bambu-принтеры парка — режим «Контроль»** ✅: камера + реле + учёт в панели, **без какой-либо интеграции с прошивкой** (никакого Developer Mode, адаптеров и обещаний). Оператор управляет ими как раньше, система их видит.
- **Marlin-принтеры — вне MVP** ✅.
- **Материалы**: ядро — PLA/PETG ✅; ABS/ASA — поддерживаются с политикой «без авто-пауз» (п. 5.3); CF-варианты — ❓ включение в программу стенда решает аудит (чем фактически печатает ФТ); TPU — вне автономии MVP ✅ (гибкую деталь сниматель не сметает предсказуемо); нейлон/PC — вне MVP.
- **Адгезия/освобождение** 🔬 — два конкурирующих направления, решаются экспериментом Э1 (ТЗ-2) с заранее заданными критериями:
  - **Ветка А: клей-автомат** — распылитель на каретке (насос, бак, автопромывка), дозированное нанесение по площади первого слоя; клей держит деталь при агрессивном обдуве (быстрый цикл) и отпускает на холодную; универсален по материалам; клей = расходник.
  - **Ветка Б: голый текстурированный PEI** — без клея вообще; пластина = расходник с учётом циклов/моек.
  - Допустимый исход — **гибрид** (PLA/PETG на PEI, ABS/CF/прочее с клеем). Состав учёта расходников адгезии в системе — по итогам Э1.

### Матрица возможностей ✅

| Функция | Автономный (Klipper + комплект «Автономия») | «Контроль» (любой принтер) |
|---|---|---|
| Телеметрия (температуры, прогресс, статус) | Moonraker WebSocket | нет — только камера |
| Старт/пауза/стоп из системы | да | нет — оператор руками |
| Назначение заданий диспетчером | автоматическое | ручная отметка занятости |
| CV-мониторинг печати | да | да |
| Чекпоинт первого слоя | да (макрос) | нет |
| Автоснятие + сортировка | да | нет |
| Реле питания / пожарный контур | да | да |
| Датчики филамента | да (штатно Klipper) | нет |

## 4. Архитектура

```
[ЦЕНТР — наш сервер, RTX 5090]
  VLM-инциденты (Qwen2.5-VL-32B AWQ, vLLM) · тяжёлая CV-перепроверка
  слайсер-воркеры (Orca CLI) · PostgreSQL · панель (React SPA/PWA) · API · аудит
        ↕ WireGuard (только исходящие соединения с площадки, без проброса портов)
[EDGE — площадка: безголовый Linux-миникомп x86]
  MQTT-брокер · КОНЕЧНЫЙ АВТОМАТ + ДИСПЕТЧЕР ОЧЕРЕДИ · зеркало очереди 24–48 ч + кэш G-code
  локальная CV (ONNX Runtime) · видеобуфер · правила деградации · управление реле
        ↕ LAN: Moonraker WS (Klipper) · MQTT (ESP32 механики) · камеры
[ПРИНТЕРЫ + периферия]
  Klipper-парк · камеры + LED-кольца · реле · дымовые датчики
  сниматель/заслонка/ИК/весы на ESP32 · [клей-модуль — по Э1]
```

**Ключевые решения:**
- **Edge = безголовый миникомп x86 (класс N100/N305/Ryzen), Linux + Docker** ✅. Требование ко ВСЕМУ железу проекта: только массовое, доступное в рознице у нескольких поставщиков; никаких нишевых компонентов. Mac mini допустим лишь как временный носитель стенда. Рабочее место оператора — любой компьютер с браузером (панель — веб).
- **Конечный автомат принтеров и диспетчер очереди исполняются на edge** ✅. Центр — надстройка (VLM, тяжёлая CV, слайсинг, панель, аналитика). Обрыв связи с центром не останавливает производственный цикл.
- **Один артефакт моделей — ONNX** ✅: ONNX Runtime с TensorRT-провайдером на центре и CPU/OpenVINO на edge; автотест паритета предсказаний на эталонном наборе кадров. Достаточность CPU-инференса edge на полном парке — ❓ подтверждает нагрузочный прогон (неделя 11); запасные ходы: миникомп с iGPU/NPU или второй узел.
- **Камеры**: стенд — USB 1080p ✅; ферма — ❓ (IP/PoE против USB-прокси-узлов; решает аудит по сети/электрике площадки и расчёт декодирования потоков).
- **Слайсер-воркер — stateless-контейнер, переносимый между центром и edge** ✅ (в MVP работает на центре; артефакты G-code едут на edge вместе с зеркалом очереди).

**Правила деградации при обрыве связи с центром** (обязательные) ✅:
1. Обрыв < 24 ч: edge продолжает очередь из кэша; локальная CV работает; авто-пауза только по локальному правилу «спагетти с confidence ≥ порога» (кроме ABS/ASA); инциденты и журнал копятся и синхронизируются после восстановления.
2. Обрыв ≥ 24 ч: допечатать текущие задания; новые не стартовать без явного локального подтверждения оператора.
3. Пожарная логика (п. 5.7) — полностью локальная и аппаратная: работает без центра И без edge.

**Стек** ✅: Python/FastAPI; PostgreSQL (партиционирование телеметрии по времени; TimescaleDB — опция роста, не требование); MQTT (Mosquitto); WireGuard; фронт — React SPA + PWA. Redis/NATS в MVP нет. Ядро не знает транспортов и каналов — интерфейсы:
- `EventBus` (сейчас Postgres+MQTT → при масштабе NATS-адаптер);
- `Notifier` (сейчас Telegram → web-push → мобильное приложение);
- `SliceService` (воркер-контейнер, см. выше).

**Центр**: Linux, Docker Compose; RTX 5090 делят vLLM, CV-перепроверка и слайсер-воркеры — единая точка отказа, **принятый риск MVP** ✅ (при недоступности центра площадка живёт по правилам деградации, VLM заменяется детерминированным fallback). Физическое размещение центра и регламент бэкапов — ❓ до конца этапа 1.

## 5. Функциональные требования

### 5.1 Реестр парка и интеграция
- **Один адаптер — Klipper/Moonraker** ✅: телеметрия, старт/пауза/стоп, исполнение макросов (`COOLDOWN_EJECT`, `WIPE`, `PARK`, `FIRST_LAYER_CHECKPOINT`), события датчиков (filament runout/motion). Bambu/Marlin-адаптеры — бэклог этапа 2+.
- Регистрация принтера: модель, объём стола, режим (автономный/«Контроль»), заряженная катушка, состояние поверхности (тип, счётчики циклов/моек) или клей-модуля (уровень, циклы с промывки) — по Э1.
- **Конечный автомат** (детерминированный, без LLM, исполняется на edge) ✅: `idle → preflight → heating → printing → cooling → ejecting → sorting → confirm_drop → idle`. Для «Контроль»-принтеров усечён до `printing/idle` по данным камеры. Каждый переход логируется.
- **Процесс «смена катушки»** (ручной ввод — MVP ✅; QR — конец MVP/этап 2 🔬): алерт «катушка кончается на #7» → оператор снимает катушку → кладёт на весы стойки (система калибрует расчётный остаток по факту) → выбирает/заводит новую катушку в панели (материал, цвет, партия) → принтер разблокирован. QR-этикетки сканируются камерой телефона через PWA — отдельное железо не нужно.

### 5.2 Библиотека изделий, очередь, диспетчер
- **Изделие — центральная сущность системы** ✅: модель (STL/3MF), версии, профили слайсинга по материалам, фото, эталоны первого слоя, статус допуска. Задание = изделие × количество.
- **Онбординг изделия**: загрузка модели → слайсинг по профилю → 3–5 калибровочных печатей под присмотром (их первые слои становятся эталонами, п. 5.3.1) → изделие получает допуски «к серии» и «к ночной смене».
- **Очередь = упорядоченный список** ✅: серийные задания, drag&drop для срочного. Поля дедлайн/приоритет в модели данных заложены, логики автопланирования в MVP нет (этап 2). Поведение диспетчера на 100% предсказуемо для оператора.
- **Диспетчер** (на edge): верхнее совместимое задание → свободный принтер (материал заряжен, стол подходит, допуски есть). Серийный режим: автоперезапуск после подтверждения сброса детали. Ступенчатый пуск нагрева пула принтеров (staggered heating) — защита проводки.
- **Preflight перед стартом**: стол пуст (CV-проверка; при неуверенности печать НЕ стартует: днём — задача оператору, ночью — принтер пропускается без будильника) ✅; температуры в норме; филамента хватает на задание; лимиты поверхности/уровень клея не превышены.

### 5.3 Видео и CV-каскад
- Камеры 1080p + LED-кольцо на принтер (постоянная освещённость — главная мера против ложных срабатываний); стол заполняет кадр.
- **Каскад**: (1) edge — лёгкая YOLO (спагетти, отлип, blob на сопле, «стол пуст/не пуст»), кадр раз в 10–30 с, при подозрении — раз в 2–5 с + серия в центр; (2) центр — крупная модель перепроверяет; (3) VLM-вердикт (п. 5.4); (4) действие.
- **Политика по материалам** ✅: PLA/PETG — авто-пауза при высоком confidence + VLM-подтверждении; ABS/ASA — **никогда не паузить автоматически** (пауза = расслоение = брак), только алерт и эскалация человеку.
- **Стартовое качество — честно** 🔬: до накопления собственного датасета (Э6 стенда: ≥300 размеченных инцидентов) детекция работает на открытых весах и ошибается чаще целевого. Catch-rate растёт с датасетом; это ожидание фиксируется и в договоре пилота.
- Каждый инцидент + вердикт + исход (подтвердил/опроверг человек) — в датасет.

#### 5.3.1 Контур первого слоя ✅ (акцент MVP)
До 70% брака закладывается в первом слое. Ключ: чекпоинт случается **один раз за печать в известный момент** — единицы событий в час на всю ферму, поэтому каждой печати достаётся тяжёлая проверка (каскад тут перевёрнут).
- **Чекпоинт**: пост-процессор G-code (п. 5.10) вставляет после первого слоя макрос — парковка головы на 2–3 с, кадр при штатном LED-свете, продолжение печати. Цена ≈15–30 с на печать; для ABS безопасно (короткая парковка, не пауза с остыванием).
- **Проверка на центре — каждая печать**: крупная YOLO + сравнение с эталонами данного изделия (методы класса PatchCore: обучение на «норме», разметка брака не нужна) + VLM при любом сомнении. До накопления эталонов изделия — VLM-only (ловит очевидное без обучения).
- **Вердикт до 5% прогресса**: `continue` / `cancel + автоперепечатка` / `эскалация человеку`. Деструктив — по правилам п. 5.4.
- Метрика contour меряется с первого дня; числовой KPI — этап 2 по данным (см. разд. 2).

### 5.4 VLM-разбор инцидентов (единственная LLM-точка MVP) ✅
- Вход: серия кадров + телеметрия + история задания. Выход: `continue | pause | cancel | escalate_human` + короткое обоснование (в журнал).
- Гард-рейлы: VLM не меняет температуры/скорости, не стартует печати; деструктивные действия (cancel) на пилоте требуют подтверждения оператора (панель/бот). Каждое решение аудируется.
- Модель: Qwen2.5-VL-32B AWQ на центре (vLLM), таймаут вердикта 30 с; при недоступности — детерминированный fallback (пороговые правила).
- **Метрика качества вердиктов**: доля вердиктов, опровергнутых человеком. Она же — критерий перевода `cancel` в автономный режим (без подтверждения) на этапе 2.

### 5.5 Трассируемость и учёт
- Журнал экземпляра детали: `экземпляр ← задание ← изделие (версия) ← слайс-артефакт (версия профиля) ← катушка (партия) ← принтер ← поверхность/клей ← исход (годен/брак/причина) ← фото финала ← вес`.
- **Журнал первичен на edge** ✅, в центр синхронизируется асинхронно и идемпотентно; «100% в журнале» ≠ «мгновенно в облаке» (см. правила деградации).
- Учёт: расход филамента (слайсер-оценка + калибровка весами при каждой замене катушки), расходники адгезии (по Э1), наработка принтеров (моточасы → регламенты ТО).
- Отчёт смены: произведено (шт/кг), брак с причинами, инциденты, загрузка, задачи оператора. Выгрузка CSV.

### 5.6 Панель и уведомления
Референс уровня UX: Creality Cloud / Bambu Handy, но вокруг **серийного производства** (библиотека изделий, а не разовые печати) ✅. Раздел строится вокруг **дня оператора**; детальная проработка — `UX-видение-панели.md` (Д).

- **Утро**: сводка ночи — что напечатано, инциденты с фото и вердиктами, задачи (забрать тару, помыть/заменить поверхность, заменить катушку, долить клей).
- **День**: постановка серий из библиотеки изделий, онбординг новых изделий (п. 5.2), замены катушек, разбор инцидентов.
- **Вечер — сценарий «Зарядить ночь»** ✅ (сердце продукта): система проверяет чек-листом — очереди хватает на 12 ч, материала хватает, расходники адгезии в норме, тары пусты, все принтеры стойки исправны → зелёная кнопка «ночь готова» / список блокеров.
- Экраны: карта парка (статусы, миниатюры камер), библиотека изделий + слайсер-поток (п. 5.10), очередь, инциденты, задачи, журнал/отчёты.
- Технология: React SPA + **PWA** ✅ (ставится на телефон/рабочий стол, web-push). Уведомления через интерфейс `Notifier`; **Telegram-бот — временный мобильный канал MVP** ✅ (алерты, подтверждение деструктива, утренняя сводка); нативное приложение — этап 2–3 по данным пилота.
- Роли: оператор / администратор фермы / наш инженер поддержки (через WireGuard).

### 5.7 Энергетика и пожарный контур (обязателен для безлюдного режима) ✅
- Управляемое реле на каждый принтер/стойку: обесточивание по команде, по дыму, по перегреву. **Только локальное управление** (MQTT/ESP-класс) — никаких реле с вендорскими облаками.
- Дымовые датчики над каждой стойкой; связка «дым → обесточивание ≤10 с» — аппаратная, работает без edge и без центра.
- Klipper: thermal runaway protection, max-температуры, watchdog — проверяются при вводе принтера в реестр.
- Staggered heating — функция диспетчера (см. 5.2).
- Порошковый модуль самосрабатывающего пожаротушения над каждой стойкой автономии (сертифицированный, автономный).
- **Регламент ежемесячной проверки пожарной цепи** (имитация дыма кнопкой теста) — задача в системе, результат со статусом «✓ ручной тест» в журнале.
- В договоре внедрения: разграничение ответственности + требования к площадке (проводка, заземление, огнетушители).

### 5.8 Механика снятия и сортировки (конструкция — ТЗ-2)
- Цикл: вердикт годен/брак **до** снятия (CV финала + история инцидентов) → заслонка-дивертер встаёт заранее → охлаждение стола с активным обдувом → **доводчик двери открывает дверь** → сметание рычагом с контролем усилия (датчик тока; превышение → стоп + алерт «деталь не снялась») → деталь по жёлобу в тару годен/брак → дверь закрывается.
- **Доводчик двери — стандартный узел каждого автономного принтера** ✅ (парк K1 весь закрытый; деталь покидает камеру через дверной проём при любом материале). Термоконтроль камеры для ABS — Э4.
- Подтверждение сброса: ИК-барьер в жёлобе (факт пролёта) + весы под тарой (приращение ≈ расчётный вес детали ±20%; заодно сигнал «тара полна»). Рассогласование → стоп принтера + алерт с фото.
- **Клей-модуль — опциональный узел комплекта** 🔬 (по Э1): распылитель на каретке, насос, бак с датчиком уровня, цикл автопромывки сопла. Требование надёжности — как у снимателя (MTBF 200 циклов). Питание — отдельная ветка с управлением через ESP32 (не с коннектора вентилятора).
- Механика (сниматель, заслонка, ИК, весы, доводчик, клей-модуль) управляется ESP32 по MQTT; контракт «edge ↔ ESP32» — в `домен-модель-и-контракты.md`.

### 5.9 Работа с филаментом
- Датчики окончания/движения филамента — штатно Klipper ✅.
- Учёт остатка: паспортный вес − слайсер-оценки расхода, **калибровка по фактическому весу снятой катушки на весах стойки при каждой замене** ✅ (ноль нового железа — весы уже в стойке).
- Диспетчер не стартует задание, если филамента не хватает; алерт «замени катушку» с привязкой к плану очереди.
- 5-кг катушки (держатели, сушка, подача) — этап 2.

### 5.10 Слайсер-ядро — в MVP ✅ (пересмотр против v1: было «этап 2»)
Урок прошлых проектов: скелет строится под продукт сразу, реализация — срезами. Поток «файл → деталь» — главный сценарий панели, он не может жить во внешнем слайсере.

**В MVP:**
- Слайсер-воркер: OrcaSlicer CLI в stateless-контейнере (вход: модель + профиль; выход: G-code-артефакт), переносим центр↔edge.
- Библиотека версионируемых профилей под связки «принтер × материал × поверхность/клей» (профили — выход Э1/Э2 стенда).
- Поток панели: загрузка STL/3MF → выбор профиля из библиотеки → серверный слайсинг → изделие в библиотеке → задание в очередь.
- **Пост-процессинг G-code, минимальный, но несущий**: вставка чекпоинта первого слоя (5.3.1) и макросов `COOLDOWN_EJECT`/`WIPE`/`PARK` по правилам. Без него не работают ни контур первого слоя, ни автономный цикл.
- Параллельный вход «загрузить готовый G-code» — всегда доступен (спасательный люк).

**Этап 2**: веб-превью слоёв, автопроверка геометрии (тонкие стенки, нависания, влезает ли), LLM-рекомендации параметров (советует — применяет человек), умный пост-процессинг (ориентация под ход снимателя, brim-якоря), клиентское имя фичи — «облачный слайсер ЛЁН».

**Лицензии** ✅: OrcaSlicer — AGPLv3, вызывается отдельным процессом (платформа не становится производной работой); патчи движка, если будут, публикуются отдельно от платформы.

## 6. Нефункциональные требования
- Автономность площадки — правила деградации (разд. 4). Edge переживает перезагрузку с автоподнятием всех сервисов ≤2 мин (systemd + Docker restart-политики).
- Надёжность: потеря кадра/сообщения не роняет конечный автомат; идемпотентность команд; повторная доставка через очередь.
- **Обновления** ✅ (новое): удалённый деплой версий на edge с автоматическим откатом при неудаче health-check; обновления только вне ночных смен.
- **Бэкапы** ✅ (новое): БД центра — ежедневно + перед миграциями; журнал edge синхронизацией в центр зарезервирован by design; регламент восстановления описывается и один раз прогоняется руками (статус «✓ ручной тест»).
- Безопасность: WireGuard-меш, без входящих портов на площадке; секреты вне кода; аудит-лог всех команд управления с инициатором (человек/правило/VLM).
- **Масштаб** 🔬: цель MVP — парк ФТ (состав уточняет аудит) + стенд. Расчётный предел архитектуры — 100 принтеров/площадка, центр — 5 площадок; подтверждается нагрузочным прогоном (80 виртуальных принтеров, неделя 11 плана), включая CPU-инференс edge.
- Наблюдаемость: Prometheus-совместимые метрики; алерт «принтер простаивает >15 мин без причины».

## 9. Приёмочные испытания (сценарии)
Каждый сценарий — чек-лист шагов; результат фиксируется в `CHANGELOG.md` со статусом «готово ✓ (ручной тест: кто, когда)».

1. **«Ночь»**: 20:00–08:00, стойка автономии, серийное задание — KPI №1, ноль вмешательств.
2. **«Первый слой»** (новый): намеренно испорченный первый слой (грязь на столе, сбитый Z-offset) ×5 — чекпоинт ловит и останавливает до 5% прогресса.
3. **«Брак»**: намеренная порча печати — полная программа ×10 на стенде (Э6); на пилоте — сокращённая, по согласованию с ФТ. На ABS — только алерты (KPI №2).
4. **«Обрыв связи»**: отключение аплинка на 4 ч посреди смены — площадка дорабатывает по правилам деградации, после восстановления журнал консистентен.
5. **«Пожарная»**: имитация дыма — обесточивание стойки ≤10 с, алерт, видеозапись сохранена.
6. **«Учёт»**: сверка журнала с физическим пересчётом тары за 3 смены — расхождение 0 (KPI №5).
7. **«Оператор»**: смена с одним человеком, все задачи из панели/бота, хронометраж (KPI №4).

## 10. Риски

| Риск | Митигация |
|---|---|
| Клеевой тракт засыхает/забивается 🔬 | цикл автопромывки; требование MTBF 200 циклов в Э1; ветка Б (PEI) как fallback |
| Текстурированный PEI деградирует 🔬 | лимиты циклов/моек в системе; ветка А (клей) как fallback; гибрид допустим |

| Стартовый CV слаб до накопления датасета | ожидания зафиксированы в ТЗ и договоре; чекпоинт первого слоя как страховка; датасет Э6 — до пилота |
| Ложные срабатывания CV ночью | LED-свет в комплекте; консервативные пороги; VLM-подтверждение; запрет авто-пауз ABS/ASA; KPI №2 |
| Edge-CPU не тянет CV полного парка | нагрузочный прогон (неделя 11); запасные ходы: узел с iGPU/NPU, второй узел |
| Один инженер на всё железо | конструкция из массовых компонентов; чертежи и документация с первого дня; стенд = копия боевого |
