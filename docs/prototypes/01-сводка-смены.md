# Прототип 01. Сводка смены

Родительский экран: `docs/UX-видение-панели.md`, раздел 5.1.

## Выбранное визуальное направление · 11.07.2026

Геометрия и плотность синхронизированы с `docs/prototypes/references/shift-summary-selected-reference.png`:
- узкая постоянная навигация;
- верхняя KPI-полоса вместо крупного hero;
- карта фермы и состояния принтеров в первом рабочем ряду;
- три компактных производственных показателя;
- журнал событий внизу;
- постоянный правый rail: следующее действие → инцидент → готовность ночи.

Две темы используют одну геометрию. Ночная тема — нейтральные темно-серые поверхности без синего подтона; светлая — белые и холодные светло-серые. Семантические цвета не окрашивают базовые фреймы.

Цель: стартовый экран оператора, который заменяет ручную проверку камер, очереди, задач и журнала. Экран должен за 10-20 секунд ответить:
- что сейчас происходит на ферме;
- что произошло за ночь;
- что требует человека;
- можно ли готовить ночной режим;
- где есть риск для выпуска.

## Формат wireframe

- Цвет: 3-4 оттенка серого, без финального брендинга.
- Разделение блоков идет через цвет поверхности, а не через border.
- Белые внешние фреймы лежат на сером фоне страницы; вложенные блоки получают светло-серую поверхность.
- Радиусы считаются по правилу `R2 = R1 - I`, где `R1` — ближайший смысловой внешний фрейм, `I` — inset до вложенного фрейма.
- Главная сетка: left nav + top status bar + центральная сводка + правая action-панель.
- В каждом блоке есть смысловая подпись, а не декоративный заголовок.
- Сначала статус и действие, потом детализация.

```text
PAGE SURFACE: серый фон

  TOP KPI SURFACE: единый фрейм, R1
    смена / время / принтеры / очередь / сервисы

  WORKSPACE SURFACE
    LEFT NAV: белый фрейм, R1

    MAIN COLUMN
      FARM OVERVIEW + PRINTER STATES: внешние фреймы, R1
        rack cells / status blocks: вложенные фреймы, R2 = R1 - I

      ROW 1
        Покрытие очереди / Выпуск / Качество: три компактных внешних фрейма

      ROW 2
        Журнал событий: табличный поток с техническими разделителями

    RIGHT ACTION PANEL: три внешних фрейма, R1
      следующее действие / инцидент / готовность ночи
```

## Состояние для тестового макета

Выбрано состояние: **утро после успешной ночи с одним нерешенным и двумя операционными блокерами**.

Причина выбора: это самый полезный первый тест формата. В нем одновременно видны выпуск, инцидент, задачи, ночные блокеры, журнал и связь edge/center.

Данные макета:
- смена: ночная завершена, дневная 08:00–20:00 активна;
- edge и center online; синхронизация журналов завершена;
- 6 автономных принтеров печатают, 1 ожидает человека, 2 control-only под камерой;
- выпущено: 26 годных деталей, 1 брак, 2,8 кг;
- инциденты: 1 ожидает human outcome по политике ABS/ASA;
- задачи: освободить тару годных деталей и заменить катушку #17;
- ночь сегодня: покрытие 18 ч 40 мин, 2 блокера.

## Блоки экрана

### A. Top status bar

Назначение: показать, можно ли доверять данным.

Содержимое:
- площадка: ФлайТех / пилот;
- текущий режим: утро, разбор ночи;
- edge: online;
- center: online;
- sync backlog: 14 записей;
- критичный safety: нет.

Действия:
- открыть состояние синхронизации;
- открыть audit log последних системных событий.

### B. Сейчас на ферме

Назначение: моментальный снимок производственного состояния.

Блоки:
- печатают: 5;
- ждут задания: 2;
- требуют человека: 3;
- control-only: 2.

Переходы:
- клик по `требуют человека` -> фильтрованные задачи;
- клик по `control-only` -> карта парка с фильтром.

### C. Выпуск смены

Назначение: коротко показать результат ночи.

Блоки:
- годен: 179;
- брак: 7;
- всего: 186;
- масса: 4.8 кг;
- выполнение очереди: 82%;
- вмешательства человека: 0.

Переходы:
- открыть отчет смены;
- открыть журнал деталей с фильтром этой смены.

### D. Инциденты

Назначение: показать, что требует решения, не скрывая историю.

Блоки:
- ждут человека: 1;
- подтверждено системой/оператором: 2;
- ложные срабатывания: 0;
- материал ABS/ASA: 1 алерт без авто-паузы.

Первое действие:
- `Разобрать инцидент #INC-0142`.

Переходы:
- открыть список инцидентов;
- открыть карточку принтера из инцидента.

### E. Готовность ночи

Назначение: показать, можно ли вечером быстро дойти до "ночь готова".

Содержимое:
- покрытие очереди: 18 ч 40 мин / 12 ч;
- блокеры: 2;
- самый дорогой блокер: заполненная тара годных на стойке A;
- preflight: 7/8 автономных принтеров готовы;
- тара: требует очистки перед продолжением `confirm_drop`.

Первое действие:
- `Открыть блокеры ночи`.

### F. Задачи оператора

Назначение: заменить обход фермы списком действий.

Приоритет:
1. Подтвердить cancel по INC-0142.
2. Заменить катушку PETG Black на K1C-07.
3. Очистить тару брака на стойке A.
4. Проверить поверхность K1C-03.

У каждой задачи:
- физический объект;
- причина;
- что разблокирует;
- кнопка перехода.

### G. Последние события

Назначение: короткий аудит без погружения в логи.

События:
- 07:58 edge синхронизирует 14 записей журнала;
- 07:42 K1C-05 confirm_drop ok;
- 07:18 VLM предложил cancel по INC-0142;
- 06:55 тара годен +18 деталей;
- 06:12 K1C-03 первый слой: continue.

## Backend handoff

Сущности:
- ShiftReport;
- Printer;
- Job;
- Incident;
- OperatorTask;
- PartInstance;
- Spool.

Команды/события:
- acknowledge incident;
- confirm cancel;
- mark task done;
- replace spool;
- mark bin emptied;
- export report;
- sync backlog status.

Минимальные поля:
- status;
- reason;
- next_action;
- physical_object;
- source: rule / CV / VLM / operator / edge / center;
- sync_state;
- updated_at;
- audit trail для действий человека.

## Acceptance для формата

- Экран читается без знания внутренней архитектуры.
- Есть ровно один главный следующий шаг: разобрать приоритетную задачу.
- Видно, что данные актуальны или частично синхронизируются.
- Видно, что это производственный экран, а не dashboard ради dashboard.
- Каждый блок можно прямо перенести в Canva как прямоугольник с подписью.
- По экрану понятно, какие backend-сущности нужны.
