# ЛЁН — правила проекта для разработчиков и AI-ассистентов

Этот файл — **единственный источник правды** по конвенциям проекта. Его читают все участники и все AI-инструменты (Claude Code, Codex/GPT, Gemini и др. — `CLAUDE.md` и `GEMINI.md` лишь указывают сюда). Правки конвенций — только здесь.

## О проекте
ЛЁН — программно-аппаратный комплекс автоматизации ферм FDM-печати: библиотека изделий → серверный слайсинг → очередь → печать на Klipper-парке → CV-контроль (включая чекпоинт первого слоя) → автоснятие → сортировка годен/брак → учёт и трассируемость. MVP — пилот на ферме ФлайТех (парк Creality K1-семейства). Команда: Ф (бэк+ML), Д (дизайн+фронт, React), И (инженер-железо).

## Дисциплина Changelog и статусов — ОБЯЗАТЕЛЬНА

Любое создание или изменение артефакта (документ, код, конструкция, модель, конфиг) фиксируется в `CHANGELOG.md` с явным статусом:

| Статус | Значение |
|---|---|
| `не начато` | работа осознана, не стартовала |
| `изучение` | исследуем подходы, решения нет |
| `в разработке` | активная работа |
| `нужен тест` | сделано, НЕ подтверждено — просить проверку |
| `готово ✓(тест)` | подтверждено тестом — указать каким: авто (имя теста) или ручной (кто, когда, что проверил) |

**Железные правила:**
1. Статус `готово` НЕ ставится без подтверждающего теста. Где возможен ручной тест — явно попросить человека проверить и описать как; только после подтверждения менять статус.
2. Всё повисшее в `нужен тест` и `в разработке` — напоминать при каждом подведении итогов; недоделки не должны тонуть.
3. AI-ассистент, завершая сессию работы, обновляет `CHANGELOG.md` и перечисляет незакрытые статусы.

## Маркировка решений в документах
- ✅ **решено** — меняется только явным новым решением (не «улучшай» молча);
- 🔬 **гипотеза** — принятое направление, проверяется указанным экспериментом;
- ❓ **открыто** — решения нет; указано, что его закрывает. Не выдумывать решение за документ.

## Принципы проекта
1. **Скелет под прод, реализация срезами**: домен-модель, контракты API и дизайн-система проектируются под полный продукт; в MVP реализуется срез. Не строить «времянку», которую придётся переделывать (урок прошлых проектов).
2. **Интерфейсы вместо технологий**: ядро не знает конкретных транспортов/каналов (`EventBus`, `Notifier`, `SliceService`). Масштабирование меняет адаптеры, не кор.
3. **Только массовое железо**: любой компонент BOM должен покупаться в рознице у нескольких поставщиков. Никакой нишевой экзотики.
4. **LLM вне контура реального времени**: детерминированное ядро (конечный автомат, диспетчер, безопасность) — код без LLM; VLM — единственная ИИ-точка (разбор инцидентов), с гард-рейлами и аудитом.
5. **Автономность площадки**: конечный автомат и диспетчер живут на edge; обрыв связи с центром не останавливает производство. Пожарная логика — аппаратная, работает без софта вообще.
6. **Один источник правды**: сроки — план этапов; конвенции — этот файл; требования — ТЗ-1. Дубли запрещены, ссылки приветствуются.
7. **Деньги — вне репозитория**: никаких цен, бюджетов, себестоимостей и финансовых договорённостей в документах и коде. Финансовые материалы живут вне репо.

## Глоссарий (минимум для входа в проект)
- **G-code** — язык команд принтера; печать = проигрывание файла команд.
- **Klipper** — современная прошивка: мозги на Linux-узле принтера, поддерживает макросы (наши `COOLDOWN_EJECT`, `FIRST_LAYER_CHECKPOINT` и др.).
- **Moonraker** — сетевой API Klipper (WebSocket); наша система — его клиент на весь парк.
- **Слайсер** — превращает 3D-модель (STL) в G-code. **OrcaSlicer** — открытый слайсер с CLI; крутится у нас на сервере как воркер. **Профиль** — набор настроек слайсинга под связку «принтер × материал × поверхность».
- **Root K1C** — официально разрешённый Creality доступ внутрь принтера → полный Moonraker.
- **Edge** — безголовый Linux-миникомп на площадке: конечный автомат, диспетчер, локальная CV, кэш очереди.
- **Центр** — наш сервер (RTX 5090): VLM, тяжёлая CV, слайсинг, панель, БД.
- **YOLO** — быстрая CV-модель («вижу спагетти, confidence 0.7»); дёшево, но простовато.
- **VLM** — мультимодальная LLM (Qwen2.5-VL): понимает контекст кадра, медленная; смотрит только эскалации и чекпоинты.
- **PatchCore/PaDiM** — методы аномалий: учатся на «норме», флажат отклонения без разметки брака (эталоны первого слоя).
- **PEI** — покрытие стола; текстурированный PEI держит деталь горячей и отпускает холодной.
- **Чекпоинт первого слоя** — вставляемая пост-процессором пауза 2–3 с после первого слоя: кадр → тяжёлая проверка каждой печати (до 70% брака закладывается в первом слое).
- **ESP32** — микроконтроллер механики (сниматель, заслонка, ИК, весы, доводчик, клей-модуль), говорит с edge по MQTT.
- **Стойка автономии** — 4–8 принтеров с полным безлюдным циклом: снятие, сортировка, учёт, пожарный модуль.
